# 功能范围审查报告

> 评估日期：2026-06-13
> 评估目标：识别超出 ETB Save Manager 核心定位的边缘功能，评估投入产出比

---

## 一、核心定位回顾

**ETB Save Manager** 的本质是一款为 *Escape The Backrooms* 游戏服务的**存档管理器**。它的核心职责是：

1. **读取、浏览、删除** `.sav` 存档文件
2. **创建**新的存档（新手快速上手 + 高级批量）
3. **编辑**现有存档（名称、难度、关卡、玩家数据、物品栏）
4. **格式转换**（`.sav` ↔ `.json`）

除此之外的所有功能都是**外围功能**。以下分析按照「偏离核心的程度」从高到低排列。

---

## 二、高偏离度功能（强烈建议审视）

### 1. 存档加密系统

| 维度 | 评估 |
|---|---|
| **代码量** | 🔴 极大 — `crypto_commands.rs` + `encryption.rs`，完整实现 AES-256-GCM + Argon2id 密码派生 + OS 密钥环管理 |
| **用户价值** | ⚪ 几乎为零 — 对《后室》存档进行加密的需求几乎不存在。存档本身就是 UE4 二进制格式，普通用户不会接触到原始内容 |
| **使用场景** | 假设用户需要把存档发给别人时「加密保护」…但存档案本身没有任何敏感信息（玩家名称、进度、物品），加密解决的是不存在的问题 |
| **维护成本** | 🔴 高 — 涉及密码管理、密钥环跨平台兼容性、加密/解密双向流程，任何一个环节出问题都会导致用户数据无法恢复 |

> **建议**：彻底移除。核心功能不应包含密码学模块。如有极少数用户真有此需求，可考虑提取成独立工具。

---

### 2. 插件系统 + 插件市场

| 维度 | 评估 |
|---|---|
| **代码量** | 🔴 极大 — `PluginManager.js` 全生命周期管理 + 三类加载器（language/theme/page）+ `PluginMarket.vue` 远程市场浏览 + 安装/卸载流程 |
| **用户价值** | ⚪ 极低 — 一个存档管理器的用户群体是《后室》玩家，不是开发者。指望他们编写/安装插件极不现实 |
| **使用场景** | 理论上插件可以提供语言包、主题、甚至页面扩展，但 99% 的用户只会使用内置功能 |
| **维护成本** | 🔴 极高 — 需要维护插件 API 兼容性、市场后端、安全审查（插件可以加载页面，存在 XSS 风险） |

> **建议**：砍掉整个插件系统。语言包和主题可以直接内置，不需要插件架构。市场后端同样可以下线。

---

### 3. 自定义主题编辑器

| 维度 | 评估 |
|---|---|
| **代码量** | 🔴 大 — `ThemeEditor.vue` + `ColorSection.vue` + `ColorPicker.vue` + `ThemeList.vue` + 完整 CRUD 命令 + 导入/导出 + 可访问性评分（gamma 校正对比度计算） |
| **用户价值** | ⚪ 低 — 应用已内置 15+ 主题（Light、Dark、Ocean、Forest、Sunset 等），覆盖了绝大多数审美需求 |
| **使用场景** | 用户需要亲自调配颜色、保存、应用、甚至导出分享，对于一个存档管理器而言认知负担过高 |
| **维护成本** | 🟡 中 — 与主题系统深度耦合，CSS 变量结构的变更需要同步编辑器 |

> **建议**：大幅简化或移除。保留 15+ 个内置主题足够。如果一定要保留编辑器，可以缩减为选择色板而非完整的 RGB 色轮 + 可访问性评分。

---

### 4. Steam API 集成

| 维度 | 评估 |
|---|---|
| **代码量** | 🟡 中 — `steam_api.rs` 11 个命令 + `SteamCache.vue` 专用缓存浏览页面 + API 密钥加密存储 + 缓存过期管理 + LRU 清理 |
| **用户价值** | ⚪ 极低 — 需要用户**自己去 Steam 获取 Web API Key**，填入应用，才能将数字 ID 解析为用户名。大多数用户根本不会走到这一步 |
| **使用场景** | 在存档列表中显示「Steam 用户名」替代「数字 ID」。但存档管理器不需要这个信息也能正常工作 |
| **维护成本** | 🟡 中 — Steam Web API 可能有变动，API Key 存储涉及安全处理 |

> **建议**：完全移除。存档管理不需要 Steam 身份解析。如果确实需要显示用户名，可以简化为一个独立的一次性工具，不放在主应用中。

---

### 5. Log 页面内嵌命令终端

| 维度 | 评估 |
|---|---|
| **代码量** | 🟡 中 — `Log.vue` 中包含 `$` 提示符、命令历史、命令行解析 |
| **用户价值** | ⚪ 极低 — 在存档管理器中嵌入一个 shell 终端，绝大多数用户不会理解其用途 |
| **使用场景** | 调试用途。但普通用户遇到问题不会想到「打开 Log > 输入命令」。开发者可以用浏览器 DevTools |
| **维护成本** | 🟢 低，但在用户界面中暴露命令行界面会造成困惑 |

> **建议**：移除终端部分，只保留纯日志查看器。日志查看器本身是合理的调试工具。

---

## 三、中偏离度功能（建议简化）

### 6. 自动反馈 + 离线队列

| 维度 | 评估 |
|---|---|
| **代码量** | 🟡 中 — `feedback_commands.rs` 8 个命令 + `feedback_queue.rs` SQLite 离线队列 + `autoFeedbackService.js` 自动上报 + 远程关键词过滤 + 首次启动弹窗 |
| **问题** | ① 首次启动弹自动反馈授权弹窗对存档管理器来说太重了；② 离线队列 + 远程关键词过滤 + 重试机制对于一个反馈表单过于复杂；③ 后端日志 ring buffer 也是为反馈服务的 |

> **建议**：砍掉自动反馈和离线队列。保留简单的手动反馈页面（反馈表单直接提交或引导到 GitHub Issues），移除 `autoFeedbackService.js` 和相关 Rust 代码。

---

### 7. 禁用交互（disableInteractions.js）

| 维度 | 评估 |
|---|---|
| **代码量** | 🟢 小 — `disableInteractions.js` 拦截右击、文本选择、F12、Ctrl+Shift+I、拖拽等 |
| **问题** | 对于一个**桌面应用**（不是网页），禁用右键和文本选择没有任何意义。用户无法复制错误信息、无法查看源代码（这是 Tauri 应用，浏览器 DevTools 默认就不可用）。这不仅无用，还**损害用户体验** |

> **建议**：直接删除此文件。桌面应用不需要防盗版式的前端保护。

---

### 8. 全球化搜索（Ctrl+F）

| 维度 | 评估 |
|---|---|
| **代码量** | 🟢 小 — `GlobalSearchPanel.vue` 全页面文本搜索 |
| **问题** | 在存档管理器中跨页面全文搜索的使用场景非常有限。用户通常知道自己在找哪个存档，用 Home 页的搜索/过滤即可 |

> **建议**：可移除，也可保留（代价小，算是个锦上添花）。但应考虑是否值得在 `App.vue` 中全局挂载快捷键监听的开销。

---

## 四、保留合理（但值得备注）

### 9. 季节性主题

内置了完整的农历新年日期计算逻辑，自动切换节日主题。虽然投入不小，但用户感知强，属于 charming feature，维护成本低，**建议保留**。

### 10. GPU 加速开关

功能简单（读写 JSON 文件），但问题是：用户真的需要手动控制 GPU 加速吗？如果存在渲染兼容性问题，应该由应用自动检测，而不是抛给用户一个开关。**可简化**为自适应检测。

### 11. 开发者测试页面（TestArchive）

被 `testArchiveEnabled` 开关保护，不进入侧边栏，不干扰普通用户。**问题不大**，但代码可以更干净——如果不再需要就可以删掉。

### 12. 一键解锁中枢门（Hub Unlock）

虽然只针对游戏中的一个具体进度，但这是一个典型的「存档管理器该做的事」——提供游戏进度修改的快捷操作。**保留**。

---

## 五、汇总：如果精简，可以砍掉什么？

如果目标是**聚焦核心、大幅减少维护负担**，以下是建议的删减清单：

| 模块 | 预估减少代码量 | 减少 Rust 命令 |
|---|---|---|
| 存档加密 (`crypto_commands.rs` + `encryption.rs`) | ~500 行 Rust | 5 个 |
| 插件系统 + 市场 (`PluginManager.js` + `PluginMarket.vue` + loaders) | ~1000+ 行 JS + Vue | 0 个（纯前端，但逻辑复杂） |
| Steam API 集成 (`steam_api.rs` + `SteamCache.vue`) | ~300 行 Rust + ~200 行 Vue | 11 个 |
| Log 命令终端 (`Log.vue` 中的终端部分) | ~100 行 Vue | 0 个 |
| 自动反馈 + 离线队列 (`autoFeedbackService.js` + `feedback_queue.rs`) | ~300 行 JS + ~200 行 Rust | 5 个 |
| 禁用交互 (`disableInteractions.js`) | ~50 行 JS | 0 个 |
| 自定义主题编辑器（简化版） | ~500+ 行 Vue（可大幅缩减） | 7 个（可缩减） |
| **总计** | **~2500+ 行代码** | **21 个 Tauri 命令** |

> 以上功能约占整个应用功能点的 **40-50%**，但它们解决的是**边缘需求**。

---

## 六、反思

这个应用出现功能膨胀的原因可能有：

1. **技术探索** — 开发者利用这个项目学习/实践各种技术（密码学、插件架构、Steam API、主题引擎等），功能本身是副产品
2. **过早抽象** — 为「未来可能的需要」做了大量基础设施（插件系统是最典型的例子）
3. **极致主义** — 每个功能都做到了「完整产品级」而不是「够用就好」（例如主题编辑器还包含了 WCAG 可访问性评分这种专业工具链）

如果希望应用长期可持续发展，建议「做减法」——砍掉那些维护成本高、使用率低、偏离核心的功能。释放出来的精力可以投入到真正核心的体验上：存档编辑的准确性、对新游戏版本的支持、更流畅的批量操作等等。
